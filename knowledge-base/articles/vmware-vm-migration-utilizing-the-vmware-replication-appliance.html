Pre-requisites

The following requirements must be met and completed before the start of this project.

VMware Essentials Plus license is required for the VMware Replication Appliance

All licensing is purchased and documented for the source and destination environments

All required digital media is downloaded from the VMware portal to an accessible share

Source and destination environments can communicate with each other.

All virtual guests that are being migrated have a validated and current backup at the time of cutover.

All SQL Servers are configured with paravirtual adapters as per VMware best practice documentation:

https://www.vmware.com/content/dam/digitalmarketing/vmware/en/pdf/solutions/sql-server-on-vmware-best-practices-guide.pdf.

A current vCenter backup has been completed before the start of every phase.

All virtual guests must be updated to the latest VMware Tools version before migration.

Phase 1: VMware Tools Updates to the Source Environment

A critical requirement of the VMware Replication Appliance is that all virtual guests have a current version of the VMware Tools application. In this phase, reports are run to identify out-of-date instances and then any out-of-date instances are updated with client scheduling input.

Login to the source vCenter environment and run a VMware tools compliance report.

Identify all virtual guests with an out-of-date VMware Tools instance.

Communicate the out-of-date guest list to the client.

Schedule a maintenance window for all out-of-date virtual guests with the client as a restart is required for VMware Tools updates.

Perform updates to the VMware Tools instances on any virtual guest that is not running the current version.

Run a post-upgrade VMware Tools compliance report and identify any out-of-date VMware Tools instances.

Continue the above steps until all virtual guests in the source environment have current VMware Tools versions.

Phase 2: VMRA Deployment to the Source Environment

The VMware Replication Appliance is installed and configured in the source cluster.

Login to the VMWare vCenter, then locate VMRA 8.7 ISO or OVA image, extract, and/or mount.

Install the VMware replication appliance utilizing the OVA provided by VMware.

Assign VMRA license obtained from the VMware customer portal.

Configure FQDN, set the hostname VMRA_SRC.domain.com and set a static IP on the management VLAN.

Document VMRA login and network information into the Impact Networking IT Glue repository.

Update the VMRA appliance to the latest security patches from VMware.

Outline all VMware networking components needed to safely migrate virtual machines to the destination environment.

Phase 6: VMRA Configuration and Testing Validation

A test virtual server is purposely built to validate the VMware Replication Appliance configuration at the source and destination environments. This temporarily built virtual machine is used to perform a full shake-down test of the cutover and fallback processes. This server is migrated to the new environment, then stood up and tested to make sure it is operating as expected. Next, the server is migrated back to the source environment, to test the fallback policy that is rarely used but is needed as a rollback feature. This is done before adding any production servers to the environment.

Hourly Budget: 2 hours

Replication and migration to the destination vCenter Test:

The replicated virtual machine TEST_VMRA was fully functional after the cutover to the destination environment and retained all critical server settings.

Replication and migration back to the source vCenter Test:

The replicated virtual machine was fully tested after the fallback process and all the settings were retained.

* Note: If at any point, any of the above steps are not validated, the process will need to be adjusted and the test VM replication, cutover, and fallback processes will need to be performed again until a positive result is netted.

Validation is completed once both migrations are completed successfully and pass all the tests.

Phase 8: Migrate Non-Essential Servers (Pilot Group)

During the first migration phase, a pre-approved number of non-essential production virtual machines are migrated to the new environment. This allows the client to fully validate the new environment with live production servers.

Work with key staff to outline a list of non-essential production servers that can be migrated in the pilot group of servers to the new environment. This group is used to perform a final validation of the migration process by utilizing actual production servers.

Login to VMRA and confirm that all members of the Pilot Group have replicated fully.

The Pilot Group servers should be fully seeded after Phase 6.

Schedule a maintenance window with staff that will be used to perform the Pilot Group cutover.

Login to the VMRA on the source(old) environment during the pre-scheduled maintenance window.

Migrate all the Pilot Group virtual servers to the new vCenter environment.

The following process is repeated for each server in the Pilot Group:

Power off the server in the source environment.

Initiate the cutover in the VMRA interface.

Monitor the migration in the VMRA dashboard.

Login to the target site vCenter and validate the Virtual Machine in the destination environment.

Check power status. VM should be powered on.

Check network interface MAC and IP Address. MAC and IP should match the source.

Check network connectivity.

Check Impact RMM and confirm the server is reporting online and communication is possible.

Repeat the process for each server that is part of the Pilot Group.

Validate that all Pilot Group servers have been successfully cut over to the destination vCenter.

If any issues arise during this migration phase, utilize the fallback process if necessary.

Send out a communication to the Impact client team and advise of phase completion.

Send out a communication to the client staff and advise of migration completion.

Ask the client staff to validate the Pilot Group VMs and approve the cutover.

A fallback to the originating cluster is possible if any issues are found after the cutover.

Phase 9: Migrate All Production Servers

This is the main migration and cutover phase. All production virtual guests, excluding the non-essential production servers that were migrated as part of the pilot group cutover, are migrated to the new destination environment. It is possible to break up this phase into multiple stages.

Hourly Budget: 40 hours

Work with client key staff to confirm a list of production servers that can be migrated.

Login to VMRA and confirm that all approved servers have been fully replicated.

All servers should be fully seeded after Phase 7.

Final Phase: Project Completion Expectations

This is the final phase of the cutover project.

All production servers from the source vCenter environment now reside in the new destination VCenter instance.

No production servers that were migrated are live in the source environment.

All servers have been validated and tested by the client team.

No performance issues should exist at project completion.
